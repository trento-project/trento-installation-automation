:toc:

== Overview

This document describes a new testing architecture for Trento installations that separates concerns between *control nodes* (where Ansible executes from) and *target nodes* (where Trento Server is installed). This approach provides more realistic testing scenarios that mirror production deployment patterns.

=== Key Changes from Current Implementation

[cols="1,1,1"]
|===
|Aspect |Current Architecture |New Architecture

|Ansible Execution
|From CI/CD runner or local machine
|From dedicated control nodes in Azure

|Node Types
|Single VM type (all-in-one)
|Control nodes + Target nodes (paired)

|Testing Scope
|Installation verification
|Full deployment + Ansible execution + verification

|SLES Version Compatibility
|Limited by runner OS
|Tests actual OS compatibility matrix
|===

=== Node Pairing Strategy

Control nodes and target nodes are paired based on OS compatibility requirements:

[cols="1,1,1"]
|===
|Control Node OS |Target Node OS |Reason

|SLES 15 SP4
|SLES 15 SP4
|Version matching

|SLES 15 SP5
|SLES 15 SP5
|Version matching

|SLES 15 SP6
|SLES 15 SP6
|Version matching

|SLES 15 SP7
|SLES 15 SP7
|Version matching

|*SLES 15 SP6*
|*SLES 16*
|Ansible not compatible on SLES 16 as control node
|===

[IMPORTANT]
====
SLES 16 cannot be used as a control node because Ansible dependencies are not yet available. We use SLES 15 SP6 as the control node for SLES 16 targets.
====

== Configuration File Format

=== .machines.pairs.csv

New configuration format defining control-target pairs:

[source,csv]
----
controlPrefix,controlSlesVersion,controlSpVersion,targetPrefix,targetSlesVersion,targetSpVersion
sles-ctrl,15,4,sles-tgt,15,4
sles-ctrl,15,5,sles-tgt,15,5
sles-ctrl,15,6,sles-tgt,15,6
sles-ctrl,15,7,sles-tgt,15,7
sles-ctrl,15,6,sles-tgt,16,0
----

*Generated VM FQDNs:*

* Control: `<controlPrefix><controlSlesVersion>sp<controlSpVersion>-ctrl.<region>.cloudapp.azure.com`
* Target: `<targetPrefix><targetSlesVersion>sp<targetSpVersion>-tgt.<region>.cloudapp.azure.com`

*Example:*

* Control: `sles-ctrl15sp6-ctrl.westeurope.cloudapp.azure.com`
* Target: `sles-tgt16sp0-tgt.westeurope.cloudapp.azure.com`

=== Advantages Over Current Approach

. *Real-world Testing*: Mimics actual production deployments where Ansible runs from control nodes
. *OS Compatibility Testing*: Validates Ansible compatibility across OS versions
. *Isolation*: Each pair is independent; failures don't cascade
. *Scalability*: Easy to add new OS version pairs
. *Automation*: Fully automated end-to-end testing
. *Cost Effective*: Control nodes can be smaller VMs (no Trento services running)

=== Potential Challenges

. *Increased VM Count*: Double the number of VMs (cost consideration)
. *Network Configuration*: Control nodes need SSH access to targets
. *Debugging*: More complex to troubleshoot (multi-hop: workflow → control → target)
. *Timing*: Sequential execution may take longer
. *Cleanup*: Need to clean up both control and target nodes

== Implementation Recommendations

=== Platform: Azure Pipelines ⭐

*Reasons:*

. *Native Azure Authentication*: Service Principal or Managed Identity eliminates credential management
. *Azure-Native Tasks*: Built-in tasks for Blob Storage, VM management, Resource Groups
. *Network Access*: Pipeline agents can be hosted in Azure, simplifying network security
. *Cost Effective*: Free tier for public repositories with generous build minutes

=== Authentication Strategy

==== Azure Pipelines Authentication

. *Service Principal* (Recommended):
+
[source,yaml]
----

  - main

  azureSubscription: 'trento-testing-sp'  # Pre-configured service connection

  - job: ProvisionAndTest
    pool:
      vmImage: 'ubuntu-latest'
    steps:
      - task: AzureCLI@2
        inputs:

=== Cost Optimization

. *VM Sizing*:
   * Control Nodes: `Standard_B2s` (2 vCPU, 4GB RAM) - ~$30/month
   * Target Nodes: `Standard_B2ms` (2 vCPU, 8GB RAM) - ~$60/month
   * Test duration: 30-45 minutes per run
   * *Cost per run*: ~$2-3 USD (spot pricing could reduce this)

. *Parallel vs Sequential*:
   * *Sequential*: Longer runtime (~90 min), lower peak cost
   * *Parallel*: Faster runtime (~45 min), higher peak cost but same total
   * *Recommendation*: Sequential for nightly tests, parallel for on-demand

. *Cleanup Strategy*:
   * Always destroy VMs after tests (even on failure)
   * Keep Storage Account and Blob Container (minimal cost)
   * Use Azure DevOps scheduled triggers for off-peak testing

== Security Considerations

. *SSH Key Management*:
   * Generate ephemeral SSH key pairs per pipeline run
   * Store in pipeline variables (not in code)
   * Destroy after cleanup

. *Secrets*:
   * SUSE registration code: Azure Key Vault or Pipeline Library
   * Blob Storage SAS token: Short-lived, read-only
   * PostgreSQL passwords: Generated per run

. *Network Security*:
   * Control nodes need SSH (22) access to targets only
   * Targets expose HTTPS (443) for verification only
   * No public SSH access to targets (only from control)

. *Service Principal Permissions*:
   * Scope to specific resource group only
   * Role: Contributor (or custom role with limited permissions)
   * Enable audit logging

== Conclusion

This new control-target architecture provides:

✅ *Higher Fidelity Testing*: Matches production deployment patterns

✅ *Better OS Coverage*: Tests Ansible compatibility across versions

✅ *SLES 16 Support*: Works around Ansible limitations

✅ *Scalability*: Easy to add new version combinations

⚠️ *Trade-offs*: Higher complexity and cost, but manageable

*Recommended Implementation*:

* Use *Azure Pipelines* for native Azure integration
* Implement *sequentially* (one pair at a time initially)
* Optimize for *cost* (destroy VMs immediately after tests)
* Monitor *closely* in first weeks to tune timing and resources

This architecture is *feasible* and provides significant value for comprehensive Trento testing. The estimated 11-16 days of development effort is justified by the improved test coverage and production-readiness validation.
